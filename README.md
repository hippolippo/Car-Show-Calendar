# 🚗 CarCalendar - Car Show Discovery Platform

> **A Learning Project**: This application was built as a hands-on exercise in **Spec-Driven Development (SDD)** and **AI-Assisted Software Engineering**. I used [SpecKit](https://speckit.dev) to design comprehensive specifications, then collaborated with AI agents to implement the features.

**Status**: MVP Core Features Complete ✅  
**Methodology**: Spec-Driven Development (SpecKit) + AI-Assisted Coding  
**Tools**: SpecKit for specifications, OpenCode/Claude for implementation

Discover car shows and meets in your area. Create events, RSVP, follow organizers, and connect with the car community.

## 🎯 Quick Start

### Prerequisites

- Node.js v22 LTS
- PostgreSQL 16+ with PostGIS extension
- Docker (recommended for PostgreSQL)

### Setup (5 minutes)

1. **Clone and Install**
   ```bash
   git clone <repository-url>
   cd CarCalendar
   
   # Backend
   cd backend
   npm install
   
   # Frontend
   cd ../frontend
   npm install
   ```

2. **Start PostgreSQL**
   ```bash
   docker run -d --name car-calendar-db \
     -e POSTGRES_DB=car_calendar \
     -e POSTGRES_USER=caruser \
     -e POSTGRES_PASSWORD=carpass123 \
     -p 5432:5432 \
     postgis/postgis:16-3.4
   ```

3. **Configure Environment**
   ```bash
   # Backend
   cd backend
   cp .env.example .env
   # Edit .env if needed
   
   # Frontend
   cd ../frontend
   cp .env.example .env
   ```

4. **Initialize Database**
   ```bash
   cd backend
   npm run migrate
   ```

5. **Start Development Servers**
   ```bash
   # Terminal 1: Backend
   cd backend
   npm run dev
   
   # Terminal 2: Frontend
   cd frontend
   npm run dev
   ```

6. **Open Application**
   - Frontend: http://localhost:5173
   - Backend API: http://localhost:3000
   - Health Check: http://localhost:3000/health

## 🚀 Development Workflow (SpecKit + AI)

### My Spec-Driven Development Process

1. **Write Feature Specification in SpecKit**
   - Define user stories and acceptance criteria
   - Specify API contracts (endpoints, request/response schemas)
   - Define functional requirements and constraints
   - Document user flows and edge cases

2. **AI Generates Implementation Plan**
   - AI designs technical architecture based on specs
   - AI creates data models (tables, relationships, constraints)
   - AI breaks down features into granular tasks
   - AI proposes technology stack and patterns

3. **Collaborate with AI Agent for Implementation**
   - AI implements features following the plan
   - I review generated code for correctness
   - AI writes tests following TDD methodology
   - Iterate based on test results and acceptance criteria

4. **Validate & Integrate**
   - Run unit tests (`npm test`)
   - Run integration tests (`npm run test:integration`)
   - Test in browser (desktop and mobile)
   - Verify against SpecKit acceptance criteria
   - Commit working features

### Test-Driven Development (TDD)
AI follows this cycle for all implementations:
1. Write tests that fail (Red)
2. Implement minimum code to pass (Green)
3. Refactor for quality (Refactor)
4. Commit changes with descriptive messages

## 🎯 What I Learned

### Spec-Driven Development with SpecKit
- **Value of Upfront Planning**: Writing detailed specs in SpecKit before coding eliminated ambiguity and rework
- **Clear Requirements = Better Code**: AI agents produce dramatically better results with precise specifications
- **API Contracts First**: Defining contracts before implementation prevents integration issues
- **Task Granularity**: Breaking features into small tasks (using SpecKit's task breakdown) makes progress measurable
- **Specification as Communication**: Specs serve as contracts between human designer and AI implementer

### AI-Assisted Development
- **AI Handles Implementation**: AI excels at architecture, boilerplate, feature implementation, and test writing
- **Humans Define Requirements**: My job was defining *what* and *why*, AI determined *how*
- **Effective Specifications**: Detailed SpecKit specs produce far better results than vague instructions
- **Iterative Refinement**: Best results come from review-feedback-iteration cycles
- **Code Review Still Essential**: AI-generated code must be validated for correctness and quality
- **AI Makes Technical Decisions**: Framework choices, patterns, and architecture were AI's domain

## 💡 Key Takeaways

**For Future Me (and Anyone Learning SDD + AI-Assisted Development):**

1. **SpecKit Makes Specifications Easy**
   - SpecKit's structure guides you to write complete, clear specifications
   - Time spent writing detailed specs pays dividends in implementation speed
   - Well-defined acceptance criteria make testing straightforward
   - AI agents work best with SpecKit-style structured specifications

2. **AI Agents Need Context, Not Control**
   - Vague prompts → mediocre code
   - Detailed SpecKit specs → high-quality implementations
   - AI makes technical decisions; humans define product requirements
   - Review and iterate for best results

3. **Let AI Handle the Technical Stuff**
   - AI chose the tech stack (Node.js, Express, PostgreSQL, React, etc.)
   - AI designed the database schema and API architecture
   - AI implemented patterns (JWT auth, REST conventions, testing frameworks)
   - I focused on *what* to build, not *how* to build it

4. **TDD Works Even Better With AI**
   - Write tests first (or have AI write them from specs), then implement
   - Tests serve as executable specifications
   - Red-Green-Refactor cycle keeps code quality high
   - AI is excellent at writing comprehensive test suites

5. **Measure Twice, Cut Once**
   - Breaking features into small, granular tasks (SpecKit's task breakdown) makes progress visible
   - Each completed task builds confidence
   - Small iterations are easier to debug than large changes
